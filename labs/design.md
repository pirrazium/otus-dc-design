# Design Decisions

## 1. Underlay Interface Model

В качестве модели underlay выбраны **routing-интерфейсы (L3)**:

* без использования trunk
* без `ip unnumbered`
* с классическими `/30` P2P сетями

### Обоснование

* Фабрика строится как **overlay-архитектура**
* Использование L2 (trunk/VLAN stretch) внутри фабрики не предполагается
* L3-only дизайн упрощает:

  * эксплуатацию
  * отладку
  * масштабирование

---

## 2. Отказ от trunk-портов

В фабрике **не используются trunk-интерфейсы**, так как:

* overlay (EVPN/VXLAN) решает задачи L2
* отсутствует необходимость растягивать VLAN’ы


---

## 3. Отказ от `ip unnumbered`

`ip unnumbered` не используется по следующим причинам:

### Ограничения

* отсутствует или ограничена поддержка:

  * iBGP
  * PIM
  * BFD

### Вывод

Так как данные технологии планируется изучать в лабе, `ip unnumbered` исключён.

---

## 4. Выбор /30 вместо /31

Используются `/30` сети для P2P линков.

### Причина

* не всё оборудование гарантированно поддерживает `/31`
* решение принято как **превентивное**, чтобы избежать проблем совместимости

---

## 5. Address Space

За основу выбран префикс:

```text
10.16.0.0/12
```

### Обоснование

* не пересекается с существующими pet-проектами
* исключает необходимость:

  * VRF для изоляции
  * NAT/overlap-обходов

---

## 6. Underlay Addressing Scheme

### Логика адресации

```text
10.<Spine-ID>.<Leaf-ID>.0/30
```

* 2-й октет — Spine ID
* 3-й октет — Leaf ID

### Пример

```text
Spine1 — Leaf1 → 10.16.101.0/30
Spine2 — Leaf1 → 10.17.101.0/30
```

---

## 7. Loopback Addressing

Для loopback-интерфейсов выделен отдельный пул:

```text
10.31.0.0/16
```

### Формат

```text
10.31.<Device-ID>.1/32
```

### Примеры

```text
Spine1 → 10.31.16.1/32
Spine2 → 10.31.17.1/32

Leaf1  → 10.31.101.1/32
Leaf3  → 10.31.103.1/32
```

---

### Дополнительные loopback’и

При необходимости:

```text
10.31.<Device-ID>.<N>/32
```

* где `<N>` — номер loopback-интерфейса

---

## 8. Подготовка под Multi-DC

Зарезервирован диапазон для второго ЦОДа:

```text
10.30.0.0/16
```

### Формат

```text
10.30.<Device-ID>.<N>
```

---

## 9. Резервирование диапазонов

### Служебные сети

```text
10.25.0.0/16 – 10.29.0.0/16 → services
```

### Зарезервировано

```text
10.24.0.0/16 → reserved
```

### Расширение лабы

```text
10.18.0.0/16 – 10.23.0.0/16 → future use
```

---

## 10. Гибкость дизайна

При необходимости:

* сервисные сети могут быть вынесены в отдельные CIDR
* допускается использование других диапазонов (включая non-RFC1918 при необходимости изоляции)

---

## 11. Итог

Принятый дизайн обеспечивает:

* простую читаемость IP-адресов
* предсказуемость схемы
* отсутствие конфликтов
* готовность к масштабированию (включая Multi-DC)
* совместимость с будущими технологиями (BGP, EVPN, VXLAN)
